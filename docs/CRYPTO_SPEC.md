# torMarketplace — Cryptographic Specification

Keys, algorithms, and on-wire formats for the backend. Browser-side crypto lives
in `wasm/src/`. See [ARCHITECTURE.md](../ARCHITECTURE.md) for the system layout.

## Key hierarchy

```
LEVEL 0 — COLD STORAGE (air-gapped, never on the server)
│
│  master_seed  [32 bytes, generated once, never persisted in plaintext]
│  • Used to derive BTC per-order owner keys
│  • Stored on disk encrypted with KEK (config.security.master_seed_hex)
│  • Held in process memory for the lifetime of the worker process
│  • Generated by `crypto::escrow::generate_master_seed()` using OsRng
│
│  worker_key   [32 bytes, generated once, independent of master_seed]
│  • Used to derive the per-order encryption key for opaque blobs
│  • Stored on disk encrypted with KEK (config.security.worker_key_hex)
│  • Held in process memory only in the worker-role process
│  • Generated by `crypto::escrow::generate_master_seed()` using OsRng
│
│  KEK          [32 bytes, KeyEncryptionKey]
│  • Encrypts master_seed_hex and worker_key_hex at rest
│  • Loaded from KEK_HEX env var ONLY in production
│  • Config-file fallback exists for dev only (deprecation warning at startup)
│  • Ephemeral mode exists for dev (EPHEMERAL_KEK=1)
│  • NEVER written to disk
│
│  server_secret  [variable length, required ≥ 32 chars]
│  • Used to derive:
│    - admin ed25519 keypair (auth_common::derive_admin_keypair)
│    - ephemeral admin identities (auth_common::derive_ephemeral_admin_identity)
│    - domain-scoped user identities (hmac_auth::derive_domain_identity)
│  • Loaded from SERVER_SECRET env var or config.server.server_secret
│  • Must be at least 32 characters (main.rs:30 enforces)
│
LEVEL 1 — SERVER MEMORY (per process)
│
│  AppState.worker_key     [u8; 32]   — worker binary only (API keeps None)
│  AppState.kek            KeyEncryptionKey  — both roles (used to decrypt keys
│                                              from config at startup)
│
│  Per-connection SOCKS5 credentials (8-slot pool)
│  • Random username suffix per slot — forces IsolateSOCKSAuth → new Tor circuit
│
│  Per-request HMAC auth key
│  • Derived: HMAC(server_secret, "auth_key_derivation_v1" || pubkey_hash)
│  • Re-derived per request, never stored
│
│  Per-request ed25519 challenge/response keypair
│  • Generated per session by the client (hypothetical WASM client)
│  • Server holds the verifying key only
│
│  Per-hour replay cache
│  • HashMap<u64_hour_bucket, HashSet<auth_token>> in stateless_auth.rs
│  • Eviction: entries older than the current hour are dropped
│
LEVEL 2 — CLIENT (WASM, not yet built)
│
│  ephemeral auth keypair (sk, pk) per request
│  ephemeral order keys (X25519) per order
│  ephemeral chat keys (X25519) per chat
│
LEVEL 3 — TRANSPORT (per-message)
│
│  HMAC auth tokens
│  Order blob (encrypted)
│  Chat blob (encrypted)
│  Dispute blob (encrypted)
```

## 3. Algorithms and parameters

### Symmetric encryption
- **Cipher:** ChaCha20-Poly1305 (RFC 8439)
- **Key:** 32 bytes, derived per-message (see above)
- **Nonce:** 12 bytes, fresh `OsRng` per encryption
- **Authentication tag:** 16 bytes (Poly1305)
- **Library:** `chacha20poly1305 = "0.10"`
- **Use sites:** `oblivious.rs:34-45` (encrypt), `oblivious.rs:47-56` (decrypt)

### Authenticated tokens
- **HMAC:** HMAC-SHA256
- **Output size:** 32 bytes
- **Library:** `hmac = "0.12"`, `sha2 = "0.10"`
- **Constant-time comparison:** `subtle::ConstantTimeEq` in `hmac_auth.rs:20`
- **Use sites:** `hmac_auth::generate_auth_token`, `auth_common::derive_auth_key`

### Key derivation
- **Per-order key:** HMAC-SHA256 (not HKDF — simpler, sufficient for this use)
- **Auth key:** HMAC-SHA256 with fixed label
- **ECDH-then-HKDF:** for WASM client blob decryption
  - HKDF-SHA256, salt = None, info = `b"tor-marketplace-order-key-v1"`
- **Library:** `hmac = "0.12"`, `hkdf = "0.12"`, `sha2 = "0.10"`

### Asymmetric
- **Ed25519:** signatures for admin keypair, ephemeral challenge/response
  - Library: `ed25519-dalek = "2"` with `pkcs8`, `pem`, `rand_core` features
- **secp256k1:** BTC owner keys, PSBT signing
  - Library: `secp256k1 = "0.29"` with `rand-std` feature
  - Library: `k256 = "0.13"` (used in some places)
  - Library: `bitcoin = "0.32"` with `rand-std` feature (PSBT construction)
- **X25519:** ECDH between WASM client and server
  - Library: `x25519-dalek = "2"` with `static_secrets` feature

### Hashing
- **BLAKE3:** general-purpose hashing
  - Library: `blake3 = "1"`
- **SHA-256:** inside HMAC and HKDF
  - Library: `sha2 = "0.10"`
- **SHA-256 also used for:** pubkey hashing in `crypto::hash::hash_pubkey`

### Password-based KDF
- **PBKDF2-HMAC-SHA256** (for KEK derivation in some paths)
  - Library: `pbkdf2 = "0.12"` with `simple` feature
  - **Iteration count:** 600,000 in frontend PBKDF2 (see `frontend/crypto.js`).

### Random number generation
- **Source:** `OsRng` (operating system CSPRNG)
- **Library:** `rand = "0.8"` provides `OsRng`
- **WASM:** `getrandom = "0.2"` with `js` feature (uses `crypto.getRandomValues`)

### Constant-time primitives
- `subtle::ConstantTimeEq` for HMAC token comparison (`hmac_auth.rs:20`)
- `crypto::zk::constant_time_compare` for byte slice comparison
- All ed25519 and secp256k1 operations are constant-time within the library
  (no custom code paths)

### Zeroization
- `zeroize = "1"` with `derive` feature
- `KeyEncryptionKey` uses `Zeroize` derive macro
- Sensitive bytes in `derive_order_key` use `StaticSecret` (which zeroizes on drop)
- Audit status: not every intermediate buffer is explicitly zeroized, but the
  libraries used (ed25519-dalek, x25519-dalek, secp256k1) do zeroize their
  internal state on drop.

## 6. File formats

### Order row in DB (4 columns)
```sql
CREATE TABLE orders (
    id BLOB PRIMARY KEY,                   -- 16 random bytes (UUIDv4)
    encrypted_order_blob BLOB,             -- opaque ChaCha20-Poly1305 ciphertext (12B nonce + N bytes)
    day_bucket INTEGER NOT NULL,           -- floor(unix_ts / 21600) * 21600
    expiry_bucket INTEGER                  -- floor(unix_ts / 21600) * 21600
);
-- plus helper columns added later: version, has_dispute, client_encrypted_blob,
-- dispute_client_blob, chat_encrypted_blob. None contain plaintext order data.
```

### Order blob (server-side encryption, `oblivious::encrypt_order_blob`)
```
[ nonce 12B | ciphertext N bytes (includes 16B Poly1305 tag) ]
```
- Key derivation: `HMAC(server_secret, "oblivious-order-v1" || order_id)` (HMAC-SHA256)
- Plaintext (today): 25-field JSON `OrderData` struct
- Plaintext: opaque client-encrypted blob (WASM convention)

### Order blob (WASM client, ECDH)
```
[ ephemeral_pk 32B | nonce 12B | ciphertext N bytes (includes 16B Poly1305 tag) ]
```
- Key derivation: `HKDF-SHA256(X25519(client_ephemeral_sk, server_static_pk), info=b"tor-marketplace-order-key-v1")`
- Server decryption: `oblivious::decrypt_ecdh_blob(blob, server_static_sk)`

### HMAC auth token
```
[ HMAC-SHA256(auth_key, "auth-token-v1" || pubkey || hour_bucket_le_u64 || path) ]
```
- 32 bytes
- Constant-time compared on server

### Admin ed25519 public key
- 32 bytes
- Derived from `server_secret`, never stored separately
